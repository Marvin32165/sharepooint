01

Présentation

SharePoint est une plateforme Microsoft qui réunit trois métiers en un : gérer les documents, bâtir un intranet et faire collaborer les équipes. Plus de 200 millions de personnes l'utilisent au quotidien.

Développée par Microsoft et intégrée à l'écosystème Microsoft 365, SharePoint sert à créer des portails intranet, à organiser les documents d'entreprise et à fournir aux équipes un espace de travail partagé. C'est aujourd'hui l'une des solutions de gestion de contenu d'entreprise les plus répandues au monde.

Plus de 20 ans d'évolution

SharePoint a suivi un mouvement de fond : partir d'un produit installé dans la salle serveur de l'entreprise pour rejoindre progressivement le cloud et ses mises à jour continues.

2001
SharePoint Portal Server 2001 — première version, centrée sur la gestion documentaire et les portails intranet basiques.
2003
SharePoint Portal Server 2003 — intégration avec Office et Active Directory, essor des sites d'équipe.
2007
Office SharePoint Server 2007 (MOSS) — arrivée des workflows, des métadonnées et d'un vrai moteur de recherche.
2010
SharePoint Server 2010 — applications de service, Business Intelligence, et première version cloud (SharePoint Online dans BPOS).
2013
SharePoint Server 2013 — modèle d'apps, API REST, recherche enrichie et lien renforcé avec Office 365.
2016
SharePoint Server 2016 — architecture hybride (Server + Online), gains de performance, rôles MinRole pour les fermes.
2019
SharePoint Server 2019 — expérience moderne (pages, listes, bibliothèques), SharePoint Framework (SPFx), OneDrive moderne.
2022
SharePoint Server Subscription Edition (SE) — modèle d'abonnement, mises à jour continues, alignement maximal avec Online.
Aujourd'hui
SharePoint Online (Microsoft 365) — évolution permanente, intégration native avec Teams, Viva Connections, Copilot et Microsoft Graph.

Ce que SharePoint cherche à résoudre

  • Gestion documentaire — centraliser, versionner et sécuriser les fichiers de l'entreprise.
  • Collaboration — faire travailler les équipes ensemble sur des sites, listes et bibliothèques partagés.
  • Portail intranet — offrir un point d'entrée unique vers les informations, actualités et ressources internes.
  • Automatisation des processus — orchestrer les validations et approbations via Power Automate.
  • Recherche d'entreprise — indexer et retrouver tout contenu, y compris dans des sources externes.
  • Intégration Microsoft 365 — s'articuler nativement avec Teams, OneDrive, Outlook, Power Platform et Copilot.

À retenir

SharePoint est la plateforme M365 de gestion documentaire et de collaboration. En vingt ans, elle est passée du portail sur site au service cloud continu — toute la suite du guide explore cette bascule.

On-site, IaaS, PaaS, SaaS

Ces quatre modèles décrivent qui gère quoi dans la pile technique. Plus on monte vers le SaaS, moins l'entreprise gère d'infrastructure — et plus elle délègue au fournisseur.

On-site — SharePoint Server

L'entreprise est responsable de tout : datacenter, réseau, serveurs, OS, SQL Server et l'application elle-même. Le modèle le plus contraignant, mais aussi le plus souverain : aucune donnée ne quitte ses murs.

IaaS — Infrastructure as a Service

Le fournisseur (Azure, AWS, GCP) fournit des machines virtuelles ; l'entreprise y installe et gère Windows Server, SQL Server et SharePoint Server. On garde le contrôle applicatif sans gérer le matériel — idéal pour porter un SharePoint Server dans le cloud sans changer de version.

PaaS — Platform as a Service

Le fournisseur gère l'OS, le runtime et le middleware ; l'entreprise ne déploie que son code et ses données. SharePoint n'a pas de mode PaaS natif, mais on s'appuie sur des services PaaS voisins (Azure App Service, Azure Functions) pour étendre SharePoint Online.

SaaS — SharePoint Online

Microsoft gère l'intégralité de la pile. L'entreprise consomme l'application via un navigateur ou les clients Microsoft 365 ; seules les données et la configuration restent sous sa responsabilité. Le modèle le plus simple à adopter et à maintenir.

À retenir

SharePoint occupe les deux extrémités du spectre : Server couvre On-site et IaaS, Online est du pur SaaS. Le PaaS n'existe pas en natif — il sert seulement à étendre Online.

05

Les couches d'abstraction

SharePoint s'empile en couches : chacune s'appuie sur celle du dessous et rend service à celle du dessus. C'est cette séparation qui permet de faire évoluer une couche sans casser les autres.

01Présentation (UI)Pages modernes & classiques · Web Parts · SPFx · Thèmes
02ApplicationSites · Listes & bibliothèques · Workflows · Power Automate
03ServicesRecherche · Métadonnées · Profils · BCS · API REST / Graph
04DonnéesSQL Server / Azure SQL · Blob Storage · Index de recherche
05InfrastructureServeurs Windows / Azure · Active Directory / Entra ID · Réseau

Présentation

Ce que voit l'utilisateur. SharePoint propose l'expérience moderne (React, responsive, intégrée à M365) et l'expérience classique (héritée). Les développeurs créent des composants sur mesure avec le SharePoint Framework (SPFx).

Application

Les objets métier : collections de sites, listes, bibliothèques et workflows. C'est ici qu'on configure les règles métier, qu'on définit les droits et qu'on branche les automatisations.

Services

Des services transversaux réutilisables : moteur de recherche (indexation full-text), métadonnées gérées (taxonomies), profils utilisateurs et Business Connectivity Services pour brancher des sources externes. L'accès programmatique passe par l'API REST ou Microsoft Graph.

Données

Tout est persisté ici. Server utilise SQL Server ; Online s'appuie sur Azure SQL et le Blob Storage pour les gros fichiers. L'index de recherche est stocké à part, pour la performance.

Infrastructure

La couche d'exécution. En Server, ce sont des serveurs Windows avec IIS, SQL Server et Active Directory. En Online, Microsoft prend tout en charge via Azure et Entra ID — la gestion devient invisible.

À retenir

Cinq couches, de l'interface à l'infrastructure. La grande différence : en Online, Microsoft absorbe tout le bas de la pile (données + infra), là où en Server vous en restez responsable.

06

Architecture physique & topologies

SharePoint Server repose sur une architecture client-serveur à trois tiers, regroupés dans une ferme. La façon de répartir ces tiers entre serveurs — la topologie — dépend des besoins de performance et de redondance.

Les trois tiers

Navigateur / client Office

↓

Web Front End (WFE) — IIS Requêtes HTTP · Authentification · Rendu des pages

↓

Serveur d'applications Recherche · Métadonnées · Profils · Workflows

↓

SQL Server — Base de données Content DB · Config DB · Service App DB · Index
  • Web Front End (WFE / IIS) — reçoit les requêtes des navigateurs et clients Office, gère l'authentification via Active Directory (NTLM, Kerberos) ou un fournisseur externe (SAML/ADFS). En production, plusieurs WFE derrière un répartiteur de charge.
  • Serveur d'applications — héberge les services (Search, Managed Metadata, User Profiles, Workflow Manager…), répartis sur plusieurs serveurs selon la charge.
  • SQL Server — cœur de la persistance : la Content DB stocke documents, listes et pages ; la Config DB contient la configuration ; les Service App DB hébergent index et profils. Always On assure la redondance.

Point important. SharePoint ne fonctionne qu'avec Microsoft SQL Server — impossible d'utiliser MySQL, PostgreSQL ou un autre moteur. Microsoft n'a jamais porté SharePoint vers d'autres bases : le couple SharePoint + SQL Server est imposé.

La ferme SharePoint

Une ferme (farm) regroupe les serveurs partageant la même base de configuration. Elle va du serveur unique (tests) à plusieurs dizaines de serveurs répartis en rôles MinRole pour la production à grande échelle.

  • Scalabilité horizontale — on ajoute des WFE ou serveurs de recherche à chaud, sans coupure.
  • Redondance — chaque rôle peut être doublé (≥ 2 WFE + SQL Always On) pour un basculement automatique.
  • Distributed Cache — service AppFabric qui met en cache flux d'activité, jetons d'authentification et aperçus.

Les 4 topologies de référence

Du plus simple au plus robuste, selon ce que l'on veut protéger et faire monter en charge.

1Tout-en-un
WFE + APP + SQL

WFE, application et SQL sur un seul serveur. Rare en production car très gourmand — réservé aux tests et démonstrations.

2SQL déporté (fréquent)
WFE + APPSQL

WFE et application sur un serveur, SQL sur un serveur dédié. La base étant l'élément le plus sensible, la stratégie de sauvegarde se concentre dessus.

3Trois rôles séparés
WFEAPPSQL

WFE, application et SQL chacun sur son serveur. Plus rare : typique d'un serveur web interne déjà chargé, où SharePoint s'ajoute sur un serveur d'applications à part.

4Scalabilité & redondance
WFE ×N (NLB)APP ×NSQL cluster

Un rôle par serveur, multiplié en nœuds et clusters. Les WFE répondent en parallèle via du NLB (répartition de charge), le SQL est mis en cluster : si un serveur tombe, un autre prend le relais. Résilience quasi totale.

Le vrai frein, c'est la licence. Techniquement, on duplique presque sans limite (plafond théorique très haut). Mais la licence SharePoint se compte par serveur et dépend de la mémoire / des cœurs : multiplier les nœuds WFE et SQL fait grimper la facture très vite. La redondance maximale n'a de sens que si la criticité la justifie.

Choisir sa topologie : l'analyse de risque

Jusqu'où faut-il redonder ? La réponse ne sort pas d'un catalogue, elle vient d'une analyse de risque : on croise la probabilité qu'une panne survienne avec son impact sur l'activité. Ce croisement dicte le niveau de redondance — et donc la topologie à retenir.

Appliqué à SharePoint : s'il ne sert qu'à stocker des photos et pièces jointes, son indisponibilité une heure ou deux n'a aucune conséquence — une simple sauvegarde suffit. Mais s'il héberge la solution métier et que la production s'arrête le temps d'une restauration (1 h–1 h 30), l'impact devient critique : il faut alors redonder (topologie 4).

Cette analyse se refait régulièrement : pour une infrastructure, un audit tous les 1 à 2 ans est la norme, car les facteurs évoluent — un serveur vieillit, une criticité change.

Online  côté Azure

En SharePoint Online, cette architecture est entièrement gérée par Microsoft sur Azure. WFE, serveurs d'applications et bases sont mutualisés et isolés par tenant (multi-locataire). L'élasticité Azure ajuste automatiquement la capacité — de façon transparente pour l'entreprise.

À retenir

Trois tiers — WFE → Application → SQL Server (obligatoire, pas d'autre base) — regroupés en ferme et déclinés en 4 topologies. Le bon niveau de redondance se décide par analyse de risque (probabilité × impact), pas par défaut. Et le point le plus sensible à sauvegarder reste toujours la base SQL.

07

Architecture logique : la hiérarchie des objets

Au-delà des serveurs physiques, SharePoint organise le contenu en une pile d'objets emboîtés, de la ferme jusqu'au site. Comprendre cette hiérarchie est indispensable pour administrer, sécuriser et sauvegarder au bon niveau.

Ferme SharePoint frontière d'administration · décrite par une seule base de configuration (Config DB)
Applications de service services mutualisés, provisionnés une fois et partagés par toutes les applications web
User Profile Recherche Managed Metadata Excel Services BCS Secure Store
Application web — IIS une URL et un mode d'authentification propres (ex. https://intranet)
Base de contenu — Content DB unité de sauvegarde & de montée en charge ; héberge des collections entières
Collection de sites frontière de sécurité & de quota — vit dans une seule base de contenu
Site de premier niveau Sous-site  ·  Sous-site → Listes · Bibliothèques · Pages
Collection de sites … une base de contenu peut en héberger plusieurs

Ferme & base de configuration

La ferme est la frontière d'administration : tout ce qui la compose est décrit par une base de configuration unique (Config DB). C'est elle qui « connaît » les serveurs, les services et les applications web. Perdre la Config DB, c'est perdre la ferme.

Applications de service

Ce sont des services partagés — recherche, profils utilisateurs, métadonnées gérées, Excel Services, Business Connectivity Services, Secure Store. On les provisionne une seule fois et plusieurs applications web les consomment : un seul service de recherche peut indexer tout l'intranet. Chaque service possède généralement sa propre base de données.

Application web

Une application web est un site IIS doté de sa propre URL et de son propre mode d'authentification. Elle s'appuie sur une ou plusieurs bases de contenu. On crée typiquement une application web distincte pour l'intranet, l'extranet ou les sites publics.

Base de contenu (Content DB)

La base de contenu est l'unité de stockage côté SQL. C'est aussi l'unité de sauvegarde et de restauration, et l'unité de montée en charge : pour répartir la charge, on ajoute des bases. Bonne pratique : surveiller leur taille (Microsoft recommande de rester raisonnable, de l'ordre de 200 Go par base pour une administration confortable).

Collection de sites & sites

Une collection de sites est la frontière de propriété, de quota et d'héritage des droits. Point clé : une collection vit entièrement dans une seule base de contenu — elle n'est jamais répartie sur plusieurs. Elle contient un site de premier niveau et, en dessous, une arborescence de sites et sous-sites qui hébergent listes, bibliothèques et pages.

À retenir

La hiérarchie logique — Ferme → Applications de service / Applications web → Content DB → Collection de sites → Sites — double l'architecture physique. Elle commande deux décisions concrètes : on sauvegarde au niveau de la base de contenu, et on sécurise au niveau de la collection de sites.

08

Pourquoi choisir SharePoint Server ?

À l'heure du tout-cloud, Server reste pertinent pour les organisations qui veulent un contrôle total sur leur infrastructure, leurs données et leurs personnalisations. Cinq usages le justifient le plus souvent.

1 · Intranet d'entreprise

Construire un portail intranet entièrement sur mesure, calé sur l'identité visuelle et les processus de l'entreprise, sans les limites du SaaS.

  • Portails thématiques — sites dédiés par département (RH, Finance, DSI…) avec actualités, annuaires, formulaires de demandes et FAQ. Chacun a son design, ses droits et ses workflows.
  • Communication centralisée — un point d'entrée unique où chacun retrouve informations stratégiques, directives et événements, sans multiplier outils et emails de diffusion.
  • Personnalisation profonde — contrairement à Online, Server autorise du développement farm-level (solutions full-trust, web parts .NET, accès direct aux bases) pour des besoins très spécifiques.

2 · Gestion documentaire

Remplacer les partages réseau (lecteurs Z:, G:…) par un système structuré, sécurisé et traçable, accessible depuis Office.

  • Versioning — à chaque modification, SharePoint conserve X versions du document. L'intérêt est concret : si vous supprimez 10 000 lignes par erreur sans vous en rendre compte et que vous enregistrez, vous revenez simplement à une version antérieure pour tout restaurer.
  • Validation par workflow — un document modifié doit être approuvé, par exemple par un supérieur : le workflow lui envoie une notification, il valide, et la nouvelle version devient disponible. Aucun échange d'email manuel.
  • Métadonnées — champs personnalisés (type, projet, statut, échéance) pour classer et retrouver précisément, là où un dossier réseau n'offre qu'un nom de fichier.
  • Permissions fines — droits définis au niveau du site, de la bibliothèque, du dossier ou du fichier ; on peut briser l'héritage pour ouvrir un seul document à un prestataire, sans exposer le reste.
  • Au-delà des lecteurs réseau — là où Z: et G: n'offrent ni versioning, ni recherche, ni workflow, SharePoint fait tout, via une liaison chiffrée entre le poste et le serveur, et reste mappable via OneDrive Sync pour qui préfère l'explorateur Windows.

3 · Automatisation des processus

Deux outils orchestrent les workflows métier sans développement lourd. SharePoint Designer reste limité au périmètre SharePoint ; Power Automate (anciennement Flow) va beaucoup plus loin : il se connecte à toute la suite Microsoft 365 et à des produits externes. C'est lui qu'on privilégie, surtout côté Online.

  • Demandes de congés — un formulaire déclenche l'envoi au manager, attend l'approbation, met à jour le planning RH et notifie l'agent, sans intervention manuelle.
  • Circuits de validation — approbations multi-étapes (achats, contrats, notes de frais) ; chaque approbateur reçoit un lien direct, et chaque décision est historisée.
  • Notifications automatiques — alertes sur modification, échéance de contrat, ajout dans une liste ou franchissement d'un seuil.
  • Synchronisation — connecteurs vers SAP, Dynamics 365, ServiceNow, SQL… ou BCS pour afficher des données externes dans les listes.

Exemple vécu — publication sur les réseaux sociaux

Une équipe de community management disposait d'une simple liste SharePoint. Chaque membre y déposait un contenu à publier ; l'ajout déclenchait automatiquement une notification au community manager. Dès qu'il validait l'entrée, le workflow récupérait le contenu et le publiait tout seul sur tous les réseaux sociaux de l'entreprise. Un seul flow remplaçait une chaîne entière d'actions manuelles.

4 · Partage des données

Un contrôle granulaire du partage, en interne comme avec l'extérieur, avec traçabilité complète.

  • Partage interne structuré — sites, sous-sites et bibliothèques avec groupes distincts (Propriétaires, Membres, Visiteurs) ; chaque service cloisonné, le commun mutualisé via des sites hub.
  • Partage externe contrôlé — inviter un prestataire sur un seul dossier via un lien sécurisé, sans ouvrir le reste de l'environnement.
  • Journalisation & audit — chaque accès, modification ou téléchargement est enregistré, exploitable pour la conformité (RGPD, ISO 27001…).
  • Intégration Active Directory — les groupes AD sont directement réutilisés ; un départ de l'entreprise révoque automatiquement tous les accès.

5 · Un espace de travail par projet et par service

SharePoint sert aussi de cadre de collaboration cloisonné : on crée un site dédié pour chaque besoin, sans mélanger les contenus.

  • Par projet — à chaque nouveau projet client, on crée un site SharePoint qui réunit toute la documentation et les outils de suivi (listes de tâches, échéances). Idéal pour la gestion de projet.
  • Par service — un espace par direction (IT, administratif, commercial avec ses offres et marchés…), chacun avec ses droits et ses processus.
  • Qualité & certification — une entreprise certifiée peut piloter sa démarche qualité dans un site dédié : circuits d'approbation des documents et archivage sécurisé, indispensables pour les audits de certification.

À retenir

Un seul fil rouge : le contrôle total. Intranet sur mesure, GED avec versioning, automatisation Power Automate, partage maîtrisé et espaces cloisonnés par projet — autant de raisons de garder la main avec Server plutôt que de déléguer à Microsoft.

Prérequis d'installation

Avant d'installer SharePoint Server (édition Subscription), quatre familles de prérequis doivent être réunies : matériel, logiciel, topologie et dépendances d'environnement. Voici l'essentiel, d'après la documentation Microsoft.

1 · Matériel

Les valeurs ci-dessous sont les minimums Microsoft. Le processeur est toujours 64 bits, 4 cœurs ; c'est la RAM qui varie selon le rôle et l'usage.

ScénarioProcesseurRAMDisque
Mono-serveur (dev / éval, services minimaux)64 bits, 4 cœurs16 Go80 Go système + 100 Go data
Mono-serveur (recette, tous les services)64 bits, 4 cœurs24 Go80 Go système + 100 Go data
Serveur en ferme multi-niveaux (dev / éval)64 bits, 4 cœurs12 Go80 Go système + 80 Go data
Serveur en ferme multi-niveaux (production, tous services)64 bits, 4 cœurs16 Go80 Go système + 80 Go data

Pratique courante. En production réelle, prévoir 32 Go de RAM ou plus par serveur d'applications (Recherche et profils utilisateurs sont gourmands), des SSD pour SQL Server, et conserver un espace libre égal à deux fois la RAM. En dessous des minimums, le support Microsoft est limité.

2 · Logiciel

  • Système d'exploitation — Windows Server 2019, 2022 ou 2025 (Standard ou Datacenter), avec interface graphique (Desktop Experience) ou en Server Core.
  • Base de données — SQL Server (Standard ou Enterprise) compatible niveau 150 : SQL Server 2019 CU5+ ou SQL Server 2022.
  • Framework — Microsoft .NET Framework 4.8.
  • Rôle serveur — Web Server (IIS) et ses composants, plus les fonctionnalités WCF / WAS.
  • Prérequis additionnels — installés automatiquement par prerequisiteinstaller.exe : .NET 4.8, Visual C++ 2015-2022 Redistributable (x64), rôle IIS et fonctionnalités associées.
  • Navigateurs — navigateurs modernes à jour (Edge, Chrome, Firefox, Safari).

Microsoft ne supporte pas l'installation d'Office et de SharePoint Server SE sur la même machine.

3 · Topologie d'installation

Deux grands choix au moment de déployer — détaillés au chapitre 06 :

  • Mono-serveur (single-server farm) — SharePoint et SQL Server sur une seule machine. Simple à monter ; adapté au développement, aux tests et aux petites charges. C'est typiquement le cas d'une VM de laboratoire.
  • Multi-serveurs (MinRole) — les rôles sont répartis sur des serveurs dédiés : Front-end, Application, Distributed Cache, Search et Custom, avec un SQL Server dédié (souvent en Always On pour la haute disponibilité). C'est la topologie de production.

4 · Autres dépendances (environnement)

  • Active Directory Domain Services — les serveurs doivent être membres d'un domaine ; les comptes de service y sont créés.
  • DNS — résolution des noms et enregistrements pour les URLs des applications web (Alternate Access Mappings).
  • Certificats SSL/TLS — pour publier les applications web en HTTPS.
  • Comptes de service dédiés, en moindre privilège :
    • Compte d'installation (setup) — administrateur local + droits SQL (dbcreator, securityadmin) ;
    • Compte de la ferme (farm account) ;
    • Comptes de pools d'applications (app pool) ;
    • Compte du service de recherche ;
    • Compte d'accès au contenu.
  • SMTP — pour les emails sortants (alertes, workflows) — optionnel.
  • Synchronisation horaire (NTP) — cohérence d'horloge entre tous les serveurs de la ferme.

À retenir

Quatre piliers avant l'installation : matériel (4 cœurs, 16–24 Go, disques séparés), logiciel (Windows Server 2019–2025 + SQL Server 2019 CU5 / 2022 + .NET 4.8), topologie (mono-serveur ou MinRole multi-serveurs) et dépendances (AD, DNS, SSL, comptes de service dédiés). Les chiffres sont des minimums — la production réclame davantage.

10

Sauvegarde des données

Le point que beaucoup découvrent trop tard : Microsoft ne garantit pas la restauration de votre contenu. Sa responsabilité porte sur la disponibilité de la plateforme — pas sur la récupération de vos fichiers après une erreur ou une attaque.

Le point à connaître

Cela ne concerne pas que SharePoint : c'est vrai pour tout Microsoft 365 — vos boîtes mail Exchange, vos fichiers OneDrive, vos espaces Teams. Vous êtes garant de vos données. Si vous perdez tout, c'est votre business et votre problème — pas celui de Microsoft.

Ce que Microsoft fait — et ne fait pas

Microsoft garantit une qualité de service élevée : le contenu est hébergé sur des datacenters redondés dans plusieurs régions, donc le service est très fiable. Mais « fiable » ne veut pas dire « sauvegardé pour vous ».

  • Fait — réplication géographique des datacenters, SLA de 99,9 %, corbeille à deux niveaux (93 jours).
  • Ne fait pas — restauration granulaire après le délai de rétention, protection contre la suppression malveillante d'une collection entière, sauvegarde vers un stockage indépendant.

Les risques réels

  • Erreur humaine — un administrateur qui supprime l'intégralité d'une boîte mail ou d'un site : le contenu part avec, et au-delà des 93 jours il est irrécupérable.
  • Attaque — un ransomware chiffre vos données ; sans sauvegarde indépendante, il n'y a rien à restaurer.
  • Souveraineté — Microsoft 365 est un produit américain. Une décision géopolitique ou une coupure de service échappe totalement à votre contrôle.

Et ce n'est pas propre à Microsoft : Google ne garantit pas davantage vos données sur Google Workspace. Sauvegarder soi-même n'est pas une obligation légale, mais c'est chaudement recommandé dans tous les cas.

Le modèle de responsabilité partagée

En SaaS, la sécurité des données et de leur sauvegarde reste à la charge de l'entreprise. Microsoft protège la plateforme, pas votre contenu métier : une stratégie de sauvegarde tierce est indispensable.

À noter : en mode Server (sur site), la question du backup « coule de source » — on sauvegarde déjà ses propres serveurs. C'est en Online que l'oubli est le plus fréquent, car on suppose à tort que « le cloud s'en occupe ».

Géré par Microsoft

  • Réplication entre datacenters
  • Sauvegarde de l'infrastructure
  • Corbeille (93 jours)
  • Versioning natif (limité)

Votre responsabilité

  • Sauvegarde externalisée des données
  • Restauration hors délai Microsoft
  • Protection contre suppression malveillante
  • Conformité & archivage long terme

Solution recommandée : Synology + Active Backup for Microsoft 365

Aujourd'hui, quasiment tous les outils de backup intègrent une brique Microsoft 365 — la sauvegarde du cloud est devenue simple. Un NAS Synology avec Active Backup for Microsoft 365 (inclus dans DSM, sans surcoût) sauvegarde automatiquement et de façon incrémentale :

  • Les bibliothèques SharePoint (fichiers, listes, pages)
  • Les boîtes Exchange Online (emails, calendriers, contacts)
  • Les espaces OneDrive de chaque utilisateur
  • Les conversations et fichiers Teams

Les sauvegardes restent sur le NAS, avec restauration granulaire (fichier, dossier, site entier) depuis une console web. La donnée reste sur un stockage que vous contrôlez — un atout pour le RGPD.

Question de contexte. Un Synology reste une solution « costaud » : une grande structure l'intègre facilement à son infrastructure, une petite épicerie un peu moins. À l'inverse, Microsoft 365 et OneDrive se déploient partout, de l'épicerie à l'entreprise nationale. Mais pour les petites structures, un simple Synology qui sauvegarde M365 fonctionne très bien — on l'installe et on n'en parle plus.

Bonnes pratiques

  • Appliquer la règle 3-2-1 : 3 copies, sur 2 supports, dont 1 hors site.
  • Tester régulièrement les restaurations — une sauvegarde jamais testée n'en est pas une.
  • Définir une RPO (perte acceptable) et une RTO (délai de restauration) adaptées à l'activité.
  • Activer le versioning SharePoint (50 versions minimum) en complément du backup.

À retenir

Microsoft garantit la disponibilité, pas la restauration de votre contenu. La sauvegarde vous incombe : un backup tiers (type Synology) et la règle 3-2-1 sont le filet de sécurité indispensable.
