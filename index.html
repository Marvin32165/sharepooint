Comprendre SharePoint — concepts, architecture et choix de déploiement

01

Présentation

SharePoint est une plateforme Microsoft qui réunit trois métiers en un : gérer les documents, bâtir un intranet et faire collaborer les équipes. Plus de 200 millions de personnes l'utilisent au quotidien.

Développée par Microsoft et intégrée à l'écosystème Microsoft 365, SharePoint sert à créer des portails intranet, à organiser les documents d'entreprise et à fournir aux équipes un espace de travail partagé. C'est aujourd'hui l'une des solutions de gestion de contenu d'entreprise les plus répandues au monde.

Plus de 20 ans d'évolution

2001
SharePoint Portal Server 2001 — première version, centrée sur la gestion documentaire et les portails intranet basiques.
2003
SharePoint Portal Server 2003 — intégration avec Office et Active Directory, essor des sites d'équipe.
2007
Office SharePoint Server 2007 (MOSS) — arrivée des workflows, des métadonnées et d'un vrai moteur de recherche.
2010
SharePoint Server 2010 — applications de service, Business Intelligence, première version cloud.
2013
SharePoint Server 2013 — modèle d'apps, API REST, recherche enrichie et lien renforcé avec Office 365.
2016
SharePoint Server 2016 — architecture hybride, gains de performance, rôles MinRole pour les fermes.
2019
SharePoint Server 2019 — expérience moderne (pages, listes, bibliothèques), SharePoint Framework (SPFx).
2022
SharePoint Server Subscription Edition (SE) — modèle d'abonnement, mises à jour continues, alignement maximal avec Online.
Aujourd'hui
SharePoint Online (Microsoft 365) — évolution permanente, intégration native avec Teams, Viva, Copilot et Microsoft Graph.

Ce que SharePoint cherche à résoudre

  • Gestion documentaire — centraliser, versionner et sécuriser les fichiers de l'entreprise.
  • Collaboration — faire travailler les équipes ensemble sur des sites, listes et bibliothèques partagés.
  • Portail intranet — offrir un point d'entrée unique vers les informations, actualités et ressources internes.
  • Automatisation des processus — orchestrer les validations et approbations via Power Automate.
  • Recherche d'entreprise — indexer et retrouver tout contenu, y compris dans des sources externes.
  • Intégration Microsoft 365 — s'articuler nativement avec Teams, OneDrive, Outlook, Power Platform et Copilot.

À retenir

SharePoint est la plateforme M365 de gestion documentaire et de collaboration. En vingt ans, elle est passée du portail sur site au service cloud continu — toute la suite du guide explore cette bascule.

On-site, IaaS, PaaS, SaaS

Ces quatre modèles décrivent qui gère quoi dans la pile technique. Plus on monte vers le SaaS, moins l'entreprise gère d'infrastructure — et plus elle délègue au fournisseur.

On-site — SharePoint Server

L'entreprise est responsable de tout : datacenter, réseau, serveurs, OS, SQL Server et l'application elle-même. Le modèle le plus contraignant, mais aussi le plus souverain : aucune donnée ne quitte ses murs.

IaaS — Infrastructure as a Service

Le fournisseur fournit des machines virtuelles ; l'entreprise y installe et gère Windows Server, SQL Server et SharePoint Server. On garde le contrôle applicatif sans gérer le matériel.

SaaS — SharePoint Online

Microsoft gère l'intégralité de la pile. L'entreprise consomme l'application via un navigateur ; seules les données et la configuration restent sous sa responsabilité.

À retenir

SharePoint occupe les deux extrémités du spectre : Server couvre On-site et IaaS, Online est du pur SaaS. Le PaaS n'existe pas en natif — il sert seulement à étendre Online.

Les couches d'abstraction

SharePoint s'empile en couches : chacune s'appuie sur celle du dessous et rend service à celle du dessus. Cette séparation permet de faire évoluer une couche sans casser les autres.

01Présentation (UI)Pages modernes & classiques · Web Parts · SPFx · Thèmes
02ApplicationSites · Listes & bibliothèques · Workflows · Power Automate
03ServicesRecherche · Métadonnées · Profils · BCS · API REST / Graph
04DonnéesSQL Server / Azure SQL · Blob Storage · Index de recherche
05InfrastructureServeurs Windows / Azure · Active Directory / Entra ID · Réseau

Présentation

Ce que voit l'utilisateur. SharePoint propose l'expérience moderne (React, responsive, intégrée à M365) et l'expérience classique (héritée). Les développeurs créent des composants sur mesure avec le SharePoint Framework (SPFx).

Services

Des services transversaux réutilisables : moteur de recherche, métadonnées gérées, profils utilisateurs et Business Connectivity Services. L'accès programmatique passe par l'API REST ou Microsoft Graph.

Données

Server utilise SQL Server ; Online s'appuie sur Azure SQL et le Blob Storage pour les gros fichiers.

À retenir

Cinq couches, de l'interface à l'infrastructure. En Online, Microsoft absorbe tout le bas de la pile (données + infra) ; en Server, vous en restez responsable.

06

Architecture physique & topologies

SharePoint Server repose sur une architecture client-serveur à trois tiers, regroupés dans une ferme. La façon de répartir ces tiers — la topologie — dépend des besoins de performance et de redondance.

Les trois tiers

Navigateur / client Office

↓

Web Front End (WFE) — IISRequêtes HTTP · Authentification · Rendu des pages

↓

Serveur d'applicationsRecherche · Métadonnées · Profils · Workflows

↓

SQL Server — Base de donnéesContent DB · Config DB · Service App DB · Index
  • Web Front End (WFE / IIS) — reçoit les requêtes, gère l'authentification via Active Directory (NTLM, Kerberos) ou SAML/ADFS.
  • Serveur d'applications — héberge les services (Search, Managed Metadata, User Profiles…).
  • SQL Server — cœur de la persistance. SharePoint ne fonctionne qu'avec Microsoft SQL Server — aucune alternative possible.

Les 4 topologies de référence

1Tout-en-un
WFE + APP + SQL

Un seul serveur. Tests et démonstrations uniquement.

2SQL déporté
WFE + APPSQL

WFE et application ensemble, SQL sur un serveur dédié. Topologie la plus courante.

3Trois rôles séparés
WFEAPPSQL

Chaque rôle sur son propre serveur.

4Scalabilité & redondance
WFE ×N (NLB)APP ×NSQL cluster

Rôles multipliés. Résilience quasi totale, mais coût de licence élevé.

À retenir

Trois tiers — WFE → Application → SQL Server (obligatoire) — déclinés en 4 topologies. Le bon niveau de redondance se décide par analyse de risque (probabilité × impact). Le point le plus sensible à sauvegarder est toujours la base SQL.

07

Architecture logique : la hiérarchie des objets

Au-delà des serveurs, SharePoint organise le contenu en une pile d'objets emboîtés, de la ferme jusqu'au site. Comprendre cette hiérarchie est indispensable pour administrer, sécuriser et sauvegarder au bon niveau.

Ferme SharePoint frontière d'administration · une seule base de configuration (Config DB)
Applications de service services mutualisés, partagés par toutes les applications web
User ProfileRechercheManaged MetadataExcel ServicesBCSSecure Store
Application web — IIS une URL et un mode d'authentification propres (ex. http://intranet)
Base de contenu — Content DB unité de sauvegarde & de montée en charge
Collection de sites frontière de sécurité & de quota — vit dans une seule base de contenu
Site de premier niveauSous-sites → Listes · Bibliothèques · Pages

Base de contenu (Content DB)

C'est l'unité de sauvegarde et de restauration. Une collection de sites vit entièrement dans une seule base de contenu — elle n'est jamais répartie sur plusieurs.

À retenir

La hiérarchie — Ferme → App. de service / App. web → Content DB → Collection de sites → Sites — commande deux décisions concrètes : on sauvegarde au niveau de la base de contenu, et on sécurise au niveau de la collection de sites.

08

Pourquoi choisir SharePoint Server ?

À l'heure du tout-cloud, Server reste pertinent pour les organisations qui veulent un contrôle total sur leur infrastructure, leurs données et leurs personnalisations.

1 · Intranet d'entreprise sur mesure

Portails thématiques par département, communication centralisée, développement farm-level (solutions full-trust, web parts .NET, accès direct aux bases) impossible en Online.

2 · Gestion documentaire avancée

  • Versioning — chaque modification conserve X versions. Une suppression accidentelle de milliers de lignes ? On revient à une version antérieure.
  • Validation par workflow — un document modifié doit être approuvé avant publication ; le workflow envoie la notification et historise chaque décision.
  • Métadonnées — champs personnalisés (type, projet, statut) pour classer et retrouver, là où un dossier réseau n'offre qu'un nom de fichier.
  • Permissions fines — droits au niveau du site, de la bibliothèque, du dossier ou du fichier ; héritage sécable à volonté.

3 · Automatisation des processus

Power Automate (anciennement Flow) est l'outil privilégié : il se connecte à toute la suite Microsoft 365 et à des produits externes. Exemples : demandes de congés, circuits de validation multi-étapes, synchronisation vers SAP ou Dynamics.

Exemple vécu — publication sur les réseaux sociaux

Une équipe de community management disposait d'une simple liste SharePoint. L'ajout d'un contenu déclenchait automatiquement une notification au community manager qui, dès validation, publiait le contenu sur tous les réseaux sociaux de l'entreprise — sans aucun échange d'email manuel.

4 · Partage des données maîtrisé

  • Partage interne structuré par groupes (Propriétaires / Membres / Visiteurs).
  • Partage externe contrôlé via lien sécurisé sur un seul dossier.
  • Journalisation complète : chaque accès, modification ou téléchargement est enregistré.
  • Intégration Active Directory : un départ révoque automatiquement tous les accès.

5 · Espaces de travail par projet et par service

Un site dédié pour chaque projet client ou service (IT, RH, commercial…), avec ses propres droits, ses listes de tâches et ses circuits d'approbation. Idéal pour la gestion de la qualité et les audits de certification.

À retenir

Un seul fil rouge : le contrôle total. Intranet sur mesure, GED avec versioning, automatisation Power Automate, partage maîtrisé et espaces cloisonnés — autant de raisons de garder la main avec Server.

2 · Logiciel

  • OS — Windows Server 2019, 2022 ou 2025 (Standard ou Datacenter).
  • Base de données — SQL Server 2019 CU5+ ou SQL Server 2022 (Standard ou Enterprise).
  • Framework — Microsoft .NET Framework 4.8.
  • Rôle serveur — Web Server (IIS) et ses composants.
  • Prérequis additionnels — installés automatiquement par prerequisiteinstaller.exe.

3 · Autres dépendances

  • Active Directory Domain Services — les serveurs doivent être membres d'un domaine.
  • DNS — résolution des noms et enregistrements pour les URLs des applications web.
  • Comptes de service dédiés en moindre privilège : setup, farm account, app pool, recherche, accès au contenu.
  • Certificats SSL/TLS — pour publier les applications web en HTTPS.

À retenir

Quatre piliers avant l'installation : matériel (4 cœurs, 16–24 Go min), logiciel (Windows Server 2019–2025 + SQL Server 2019 CU5 / 2022 + .NET 4.8), topologie (mono-serveur ou MinRole) et dépendances (AD, DNS, SSL, comptes de service dédiés).

10

Sauvegarde des données

Le point que beaucoup découvrent trop tard : Microsoft ne garantit pas la restauration de votre contenu. Sa responsabilité porte sur la disponibilité de la plateforme — pas sur la récupération de vos fichiers.

Le point à connaître

Cela concerne tout Microsoft 365 : vos boîtes Exchange, vos fichiers OneDrive, vos espaces Teams. Vous êtes garant de vos données. Si vous perdez tout, c'est votre business et votre problème — pas celui de Microsoft.

Ce que Microsoft fait — et ne fait pas

  • Fait — réplication géographique des datacenters, SLA 99,9 %, corbeille à deux niveaux (93 jours).
  • Ne fait pas — restauration granulaire après le délai, protection contre suppression malveillante d'une collection entière, backup vers un stockage indépendant.

Le modèle de responsabilité partagée

Géré par Microsoft

  • Réplication entre datacenters
  • Sauvegarde de l'infrastructure
  • Corbeille (93 jours)
  • Versioning natif (limité)

Votre responsabilité

  • Sauvegarde externalisée des données
  • Restauration hors délai Microsoft
  • Protection contre suppression malveillante
  • Conformité & archivage long terme

Solution recommandée : Synology + Active Backup for M365

Un NAS Synology avec Active Backup for Microsoft 365 (inclus dans DSM, sans surcoût) sauvegarde automatiquement et de façon incrémentale les bibliothèques SharePoint, les boîtes Exchange Online, les espaces OneDrive et Teams. Restauration granulaire depuis une console web. La donnée reste sur un stockage que vous contrôlez — un atout pour le RGPD.

Bonnes pratiques

  • Appliquer la règle 3-2-1 : 3 copies, sur 2 supports, dont 1 hors site.
  • Tester régulièrement les restaurations — une sauvegarde jamais testée n'en est pas une.
  • Définir une RPO (perte acceptable) et une RTO (délai de restauration) adaptées à l'activité.

À retenir

Microsoft garantit la disponibilité, pas la restauration de votre contenu. La sauvegarde vous incombe : un backup tiers (type Synology) et la règle 3-2-1 sont le filet de sécurité indispensable.

P01

Préparer SQL Server 2022 pour SharePoint

SharePoint Server ne fonctionne qu'avec Microsoft SQL Server. Cette étape se réalise sur SRV-SQL01 (172.16.0.3), joint au domaine orion.local, avant toute installation SharePoint.

Comptes de service à créer (sur SRV-AD01)

Avant l'installation, créer ces quatre comptes dans Active Directory — Users and Computers → OU de votre choix :

CompteRôleDroits SQL nécessaires
orion\sp_farmCompte de ferme SharePointdbcreator, securityadmin
orion\sp_setupInstallation et admin localdbcreator, securityadmin
orion\sp_servicesApp pools des applications webAucun (SharePoint les attribue)
orion\sp_searchService de rechercheAucun (SharePoint les attribue)

Ajouter orion\sp_setup en tant qu'administrateur local sur SRV-Sharepooint01 (via GPO ou Local Users and Groups).

Installation de SQL Server 2022

  1. 01
    Lancer le setup SQL Server 2022

    Monter l'ISO SQL Server 2022, exécuter setup.exe en administrateur. Choisir Installation → New SQL Server stand-alone installation.

  2. 02
    Sélectionner les fonctionnalités

    Cocher uniquement Database Engine Services. Les outils de management (SSMS) s'installent séparément.

  3. 03
    Configurer la collation — étape critique

    Dans Server Configuration → Collation, cliquer Customize et saisir exactement : SQL_Latin1_General_CP1_CI_AS. SharePoint refusera de s'installer avec toute autre collation.

  4. 04
    Configurer l'authentification et les admins SQL

    Choisir Windows Authentication Mode. Dans SQL Server Administrators, ajouter orion\sp_farm et orion\sp_setup.

  5. 05
    Activer le protocole TCP/IP

    Ouvrir SQL Server Configuration Manager → SQL Server Network Configuration → Protocols for MSSQLSERVER → activer TCP/IP. Redémarrer le service SQL Server.

  6. 06
    Ouvrir le pare-feu Windows

    Créer une règle entrante autorisant TCP 1433 (port SQL Server par défaut).

    PowerShell — SRV-SQL01
    New-NetFirewallRule -DisplayName "SQL Server" `
    -Direction Inbound -Protocol TCP `
    -LocalPort 1433 -Action Allow
  7. 07
    Limiter la mémoire SQL Server

    Dans SSMS, clic droit sur le serveur → Properties → Memory. Définir Maximum server memory à RAM totale − 2 Go (ex. : 14 336 Mo pour 16 Go de RAM). Évite que SQL Server n'affame l'OS.

Bonne pratique

Ne jamais utiliser le compte sa pour SharePoint. Windows Authentication est plus sécurisé et recommandé par Microsoft. Mettre les fichiers de données (.mdf), de journaux (.ldf) et tempdb sur des disques séparés si possible.

P02

Installer SharePoint Server SE sur Windows Server 2025

Installation complète sur SRV-Sharepooint01 (172.16.0.2), joint à orion.local. Toutes les commandes se font en étant connecté en tant que orion\sp_setup (admin local + droits SQL).

Vérifications avant de démarrer

  • La VM est jointe au domaine orion.local et a redémarré.
  • IP statique : 172.16.0.2 ; DNS primaire : 172.16.0.1 (SRV-AD01).
  • SRV-SQL01 est accessible : Test-NetConnection -ComputerName SRV-SQL01 -Port 1433 doit répondre TcpTestSucceeded : True.
  • IPv6 désactivé sur la carte réseau.

Étapes d'installation

  1. 01
    Monter l'ISO SharePoint Server SE et lancer les prérequis

    Exécuter prerequisiteinstaller.exe en administrateur. Il installe automatiquement .NET Framework 4.8, Visual C++ Redistributable, WCF Activation, et configure le rôle IIS avec tous les composants requis. Le serveur peut redémarrer 2 à 3 fois — laisser le processus se terminer jusqu'au message Installation complete.

    Sans accès internet

    Utiliser le switch /unattended avec les chemins locaux vers chaque prérequis. Voir la documentation Microsoft pour la liste complète des packages offline.

  2. 02
    Lancer setup.exe

    Exécuter setup.exe en administrateur. Entrer la clé produit (ou laisser vide pour la version d'évaluation 180 jours). Accepter le CLUF → sélectionner le chemin d'installation → Install Now. Durée : 10 à 20 minutes. À la fin, décocher l'option "Run the SharePoint Products Configuration Wizard now" avant de fermer.

  3. 03
    Exécuter l'assistant de configuration de ferme (PSConfig)

    Démarrer → Microsoft SharePoint Products → SharePoint Products Configuration Wizard (en administrateur). Répondre aux écrans :

    • Connect to a server farm → Create a new server farm
    • Database server → SRV-SQL01
    • Database name → SP_Config (laisser par défaut)
    • Username → orion\sp_farm + mot de passe
    • Passphrase → choisir une passphrase forte et la noter précieusement — indispensable pour ajouter d'autres serveurs à la ferme.
    • Central Administration Web Application port → 2016
    • Authentication provider → NTLM

    Cliquer Next, puis Finish. La configuration exécute 10 tâches (3 à 8 minutes).

  4. 04
    Accéder à l'Administration centrale

    Ouvrir un navigateur sur SRV-Sharepooint01 (ou depuis le poste hôte via 172.16.0.2) :

    Navigateur
    http://SRV-Sharepooint01:2016

    Se connecter avec orion\sp_setup. La page d'accueil de l'Administration centrale confirme que l'installation est réussie.

Bonne pratique

Ne jamais installer SharePoint sous un compte Domain Admin — utiliser sp_setup avec uniquement les droits nécessaires. Conserver la passphrase de ferme dans un gestionnaire de mots de passe : sans elle, il est impossible d'ajouter un serveur à la ferme ultérieurement.

P03

Configurer la ferme

La ferme est créée, mais les services sont vides. Cette étape configure les comptes gérés, démarre les services essentiels, crée les applications de service et prépare le DNS pour accéder au futur intranet.

  1. 01
    Enregistrer les comptes gérés (Managed Accounts)

    Dans l'Administration centrale : Security → General Security → Configure Managed Accounts → Register Managed Account. Enregistrer orion\sp_services et orion\sp_search. SharePoint peut ainsi gérer les rotations de mots de passe automatiquement.

  2. 02
    Démarrer les services sur le serveur

    System Settings → Manage Services on Server. Démarrer au minimum :

    • Microsoft SharePoint Foundation Web Application
    • SharePoint Server Search
    • User Profile Service (si profils AD requis)
  3. 03
    Créer les applications de service

    Application Management → Manage Service Applications → New. Créer dans cet ordre :

    • Managed Metadata Service — taxonomies partagées entre collections de sites.
    • Search Service Application — moteur d'indexation full-text. Compte de service : orion\sp_search.
    • User Profile Service Application — synchronisation des profils avec Active Directory.

    Alternativement : Home → Farm Configuration Wizard crée automatiquement les applications de service de base en quelques clics.

  4. 04
    Créer l'enregistrement DNS pour l'intranet

    Sur SRV-AD01, ouvrir DNS Manager → zone de recherche directe orion.local → Nouveau Hôte (A) :

    PowerShell — SRV-AD01
    Add-DnsServerResourceRecordA -ZoneName "orion.local" `
    -Name "intranet" -IPv4Address "172.16.0.2"

    L'URL http://intranet sera résolu vers SRV-Sharepooint01.

Bonne pratique

Configurer les applications de service manuellement plutôt qu'avec l'assistant automatique : c'est plus long, mais vous comprenez exactement ce que vous créez et pouvez choisir les comptes de service appropriés pour chaque service.

P04

Créer son premier site

La ferme est opérationnelle. Il faut maintenant créer une application web (le conteneur IIS), puis une collection de sites (le premier espace utilisable), et enfin configurer une bibliothèque de documents.

  1. 01
    Créer l'application web

    Central Admin → Application Management → Manage Web Applications → New. Paramètres :

    • IIS Web Site Name : SharePoint - 80
    • Port : 80 — URL : http://intranet
    • Application Pool : créer nouveau → nom SharePoint Web Services Default → compte orion\sp_services
    • Database Server : SRV-SQL01 — Database Name : SP_Content_Intranet

    Cliquer OK (1 à 2 minutes).

  2. 02
    Créer la collection de sites

    Application Management → Create Site Collections. Paramètres :

    • Web Application : http://intranet
    • Title : Intranet Orion
    • URL : racine /
    • Template : Team Site (onglet Collaboration)
    • Primary Site Collection Administrator : orion\sp_setup

    Cliquer OK.

  3. 03
    Accéder au site et créer une bibliothèque

    Ouvrir http://intranet dans un navigateur. Se connecter en tant que orion\sp_setup. Le site d'équipe s'affiche. Aller dans Site Contents → New → Document Library, nommer la bibliothèque Procédures IT et cliquer Create.

  4. 04
    Ajouter des métadonnées et tester le versioning

    Dans les paramètres de la bibliothèque (Library Settings) : activer le versioning (major versions), ajouter une colonne de choix Statut (En cours / Validé / Archivé). Uploader un document, le modifier, vérifier l'historique des versions dans Version History.

Bonne pratique

Créer une application web distincte par usage (intranet, extranet, sites publics) plutôt que tout mettre dans la même. Chaque application web a son propre app pool, son mode d'authentification et ses bases de contenu — la séparation limite les risques et facilite la sauvegarde.

P05

Gérer les droits et les permissions

SharePoint propose un modèle de permissions granulaire : des niveaux de permission (ensembles d'actions autorisées) assignés à des groupes à différents niveaux de la hiérarchie (site, bibliothèque, dossier, document).

Groupes SharePoint par défaut

GroupeNiveau de permissionCe que l'utilisateur peut faire
Intranet Orion OwnersFull ControlTout faire, y compris gérer les permissions et supprimer le site
Intranet Orion MembersEdit / ContributeAjouter, modifier et supprimer des documents et éléments de listes
Intranet Orion VisitorsReadLire uniquement — aucune modification possible

Ajouter des utilisateurs ou groupes AD

  1. 01
    Via l'interface SharePoint

    Site Settings → People and Groups → Intranet Orion Members → New → Add Users. Saisir orion\Domain Users (ou le groupe AD souhaité) et valider. Le groupe AD est directement réutilisé — pas besoin de recréer les utilisateurs dans SharePoint.

  2. 02
    Couper l'héritage sur une bibliothèque

    Dans Library Settings → Permissions for this document library → Stop Inheriting Permissions. Retirer le groupe Members, ajouter uniquement les utilisateurs autorisés. Idéal pour cloisonner un dossier confidentiel (RH, contrats) tout en conservant les autres bibliothèques accessibles à tous.

  3. 03
    Créer un niveau de permission personnalisé

    Site Settings → Site Permissions → Permission Levels → Add a Permission Level. Exemple : niveau "Lecture + Ajout uniquement" — cocher View Items, Add Items, sans Edit ou Delete. Utile pour des utilisateurs qui soumettent des documents sans pouvoir modifier l'existant.

Bonne pratique

Toujours assigner des permissions à des groupes, jamais à des utilisateurs individuels. Lorsqu'un collaborateur quitte l'entreprise, il suffit de désactiver son compte AD : SharePoint révoque tous ses accès automatiquement, sans avoir à parcourir chaque site et bibliothèque manuellement.

P06

Automatiser avec SharePoint Designer

Sur un lab SharePoint Server SE sans connexion internet, SharePoint Designer 2013 est l'outil d'automatisation natif disponible gratuitement. Il permet de créer des workflows déclenchés par des événements sur les listes et bibliothèques.

Installer SharePoint Designer 2013

Télécharger SharePoint Designer 2013 (gratuit, version x64) depuis le centre de téléchargement Microsoft. L'installer sur le poste d'administration ou directement sur SRV-Sharepooint01. La version 2013 est compatible avec SharePoint Server SE en mode classique.

Créer un workflow d'approbation

  1. 01
    Ouvrir le site dans SharePoint Designer

    Lancer SharePoint Designer → Open Site → entrer l'URL http://intranet. Se connecter avec orion\sp_setup. Le site se charge dans le volet gauche.

  2. 02
    Créer le workflow

    Workflows → List Workflow → sélectionner la liste ou bibliothèque cible (ex. "Procédures IT") → nommer le workflow "Approbation Procédure". Choisir la plateforme SharePoint 2013 Workflow.

  3. 03
    Ajouter les actions

    Dans le Designer de workflow, ajouter les actions :

    • Set Field in Current Item → champ Statut = "En attente d'approbation"
    • Send an Email → configurer le destinataire (responsable), l'objet et le corps du message avec des champs dynamiques (Current Item:Title)
    • Start Approval Process → désigner l'approbateur et le délai
  4. 04
    Configurer le déclencheur et publier

    Dans les Workflow Settings, cocher Start workflow automatically when an item is created. Cliquer Publish. Le workflow est actif : uploader un document dans "Procédures IT" pour le tester.

Pour aller plus loin — Power Automate

En production hybride (SharePoint Server SE connecté à Microsoft 365), Power Automate remplace avantageusement SharePoint Designer : il supporte des centaines de connecteurs (SAP, Salesforce, Dynamics…), propose des approbations modernes sur mobile et s'administre via une interface web. Il requiert une licence Microsoft 365 et la passerelle de données on-premises.

P07

Administrer SharePoint via PowerShell

Deux outils PowerShell coexistent : le SharePoint Management Shell (classique, natif sur le serveur) et PnP PowerShell (moderne, multiplateforme). Les deux s'utilisent quotidiennement en administration.

SharePoint Management Shell

À lancer sur SRV-Sharepooint01 via le menu Démarrer → Microsoft SharePoint Products → SharePoint Management Shell (en administrateur). Doit être exécuté avec un compte administrateur de ferme.

SharePoint Management Shell
# Informations sur la ferme
Get-SPFarm | Select BuildVersion, Status

# Lister les applications web
Get-SPWebApplication | Select Url, Name

# Lister toutes les collections de sites
Get-SPSite -Limit All | Select Url, ContentDatabase

# Créer une nouvelle collection de sites
New-SPSite -Url "http://intranet/sites/rh" `
    -Name "Ressources Humaines" `
    -Template "STS#0" `
    -OwnerAlias "orion\sp_setup"

# Lister les listes d'un site
$web = Get-SPWeb "http://intranet"
$web.Lists | Select Title, ItemCount
$web.Dispose()

# Sauvegarder la ferme complète
Backup-SPFarm -Directory "\\SRV-SQL01\Backup\SP" `
    -BackupMethod Full -Verbose

PnP PowerShell

Module moderne installable sur n'importe quelle machine avec PowerShell 7+. Plus expressif que le Management Shell pour les opérations sur le contenu.

PowerShell 7 — n'importe quelle machine
# Installation du module
Install-Module PnP.PowerShell -Scope CurrentUser

# Connexion à SharePoint Server on-premises
Connect-PnPOnline -Url "http://intranet" -CurrentCredentials

# Lister toutes les listes
Get-PnPList | Select Title, ItemCount, Hidden

# Récupérer les éléments d'une liste
Get-PnPListItem -List "Procédures IT" | Select Id, FieldValues

# Uploader un fichier
Add-PnPFile -Path "C:\docs\procedure.docx" `
    -Folder "Procédures IT"

# Ajouter un utilisateur à un groupe
Add-PnPGroupMember `
    -LoginName "orion\jdupont" `
    -Identity "Intranet Orion Members"

# Créer un sous-site
New-PnPWeb -Title "Projets" -Url "projets" `
    -Template "STS#0" -Locale 1036

Bonne pratique

Scripter toutes les opérations répétitives (création de sites, attribution de droits, provisionnement de bibliothèques) plutôt que de les faire via l'interface. Les scripts deviennent la documentation vivante de l'environnement et permettent de reconstruire la configuration rapidement après un incident.

P08

Sauvegarder et restaurer SharePoint Server

Une stratégie de backup SharePoint Server SE s'articule sur trois niveaux complémentaires : la ferme complète, les bases de contenu SQL, et les collections de sites individuelles. Chaque niveau répond à un besoin de restauration différent.

Niveau 1 — Backup de la ferme complète

Couvre la configuration, les applications de service et toutes les bases de contenu. Utile pour reconstruire un environnement entier.

SharePoint Management Shell
# Backup complet de la ferme
Backup-SPFarm -Directory "E:\Backups\SP\$(Get-Date -f yyyyMMdd)" `
    -BackupMethod Full -Verbose

# Backup différentiel (plus rapide, nécessite un full préalable)
Backup-SPFarm -Directory "E:\Backups\SP\$(Get-Date -f yyyyMMdd)" `
    -BackupMethod Differential

Niveau 2 — Backup SQL des bases de contenu

La méthode la plus rapide et la plus fiable. Se fait directement via SQL Server Management Studio ou PowerShell sur SRV-SQL01.

PowerShell — SRV-SQL01
# Importer le module SQL Server
Import-Module SqlServer

# Backup de la base de configuration (toujours la sauvegarder)
Backup-SqlDatabase -ServerInstance "SRV-SQL01" `
    -Database "SP_Config" `
    -BackupFile "E:\Backups\SQL\SP_Config_$(Get-Date -f yyyyMMdd).bak"

# Backup de la base de contenu intranet
Backup-SqlDatabase -ServerInstance "SRV-SQL01" `
    -Database "SP_Content_Intranet" `
    -BackupFile "E:\Backups\SQL\SP_Content_Intranet_$(Get-Date -f yyyyMMdd).bak"

Niveau 3 — Export d'une collection de sites

Granularité maximale : exporter une seule collection, ses sous-sites, sa sécurité et les versions de documents.

SharePoint Management Shell
# Export d'une collection de sites
Export-SPWeb -Identity "http://intranet" `
    -Path "E:\Backups\Sites\intranet_$(Get-Date -f yyyyMMdd).cmp" `
    -IncludeUserSecurity -Versions 4 -Verbose

# Restauration d'une collection dans une application web existante
Import-SPWeb -Identity "http://intranet/sites/restore-test" `
    -Path "E:\Backups\Sites\intranet_20260630.cmp" -Force

# Monter une base de contenu SQL restaurée
Mount-SPContentDatabase -Name "SP_Content_Intranet" `
    -WebApplication "http://intranet" `
    -DatabaseServer "SRV-SQL01"

Planifier les sauvegardes automatiques

Sur Windows Server 2025, créer une tâche planifiée via le Task Scheduler ou une règle Windows Server Backup qui déclenche le script PowerShell de backup chaque nuit. Stocker les sauvegardes sur un partage réseau (\\SRV-AD01\Backup) pour respecter la règle 3-2-1 : donnée locale + partage réseau + NAS Synology hors site.

Bonne pratique

Toujours tester la restauration avant d'en avoir besoin : monter une base de contenu restaurée sur un port alternatif (ex. http://intranet:8080) et vérifier que les données sont récupérables. Une sauvegarde non testée n'est qu'une hypothèse. Planifier ce test au minimum une fois par trimestre.

À retenir

Trois niveaux de backup : ferme complète (Backup-SPFarm), bases SQL (le plus fiable, Backup-SqlDatabase) et collection de sites (Export-SPWeb). La règle 3-2-1 reste la norme. Tester les restaurations régulièrement — une sauvegarde jamais testée n'est pas une sauvegarde.
